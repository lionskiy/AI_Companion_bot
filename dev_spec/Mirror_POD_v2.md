## 1. Видение продукта

### 1.1 Концепция

«Зеркало» — персонализированный AI-компаньон, который помогает пользователю познать себя через сочетание психологии, астрологии, таро, нумерологии и толкования снов. Продукт работает в формате живого текстового диалога через Telegram, MAX, VK, WhatsApp, мобильное приложение и веб, и помнит пользователя на протяжении всего времени использования.

Компаньон не просто реагирует на сообщения — он **живёт**: инициирует разговоры, создаёт ощущение живого присутствия, помнит детали, проявляет характер. Пользователь не забывает о нём среди сотен других чатов.

> **Главная идея:** не предсказания будущего, а инструмент рефлексии и самопознания. Компаньон говорит языком астрологии и психологии, но решает реальную потребность — быть услышанным, понятым и получить персональный отклик.

### 1.2 Позиционирование

| Аудитория | Возраст | Потребность |
|-----------|---------|------------|
| Женщины, саморазвитие | 22–40 лет | Понять себя, отношения, предназначение |
| Мужчины, психология | 25–45 лет | Анонимная рефлексия, рост, карьера |
| Молодёжь, эзотерика | 18–28 лет | Развлечение, астрология, идентичность |
| 18+ аудитория | 18+ | Эмоциональная близость, флирт, ролевые игры |

**Ключевая боль:** качественные консультации (живой астролог: 3 000–15 000 руб./сессия) недоступны ежедневно. «Зеркало» заполняет этот пробел персонализированным AI-диалогом за 299–999 руб./мес.

## 3. Функциональная карта (Feature Map)

### 3.1 Режимы диалога (Intent Router)

| Режим | Триггер | Что делает |
|-------|---------|------------|
| **Ежедневный ритуал** | Утреннее сообщение / /start | Карта дня (таро), транзит дня, намерение, вопрос для рефлексии |
| **Астрология** | Вопрос о гороскопе, планетах, домах | Расчёт Swiss Ephemeris + RAG-интерпретация |
| **Таро** | «Раскинь карты», «Расклад» | Виртуальная колода 78 карт, RAG по значениям позиций |
| **Психология** | Вопрос о себе, отношениях, страхах | CBT/ACT техники, нарративная рефлексия |
| **Предназначение** | «Не знаю чего хочу», карьера | Работа с ценностями, архетипами, картой жизни |
| **Сонник** | «Мне приснилось...» | RAG по базе символов + астро-контекст даты сна |
| **Совместимость** | «Мы с...», синастрия | Сравнение двух натальных карт, советы |
| **Нумерология** | Дата, имя, числа | Расчёт числа судьбы, года, имени + интерпретация |
| **Дневник** | Вечерняя рефлексия | Запрос от бота + сохранение записи в память |
| **Квест/Игра** | Команда /play или меню игр | Адаптивная текстовая игра с психологическим финалом |
| **18+ (Adult Mode)** | Доступ только при верифицированном возрасте | Флирт, ролевые игры, откровенное общение |
| **Проактивный** | Инициируется ботом | Бот пишет первым, имитирует жизнь, вовлекает |

### 3.2 Астрология (детально)

- **Натальная карта:** Солнце, Луна, 8 планет + Лилит, Хирон, Узлы в знаках и домах (системы: Плацидус, Равнодомная, Кох — настраивается)
- **Аспекты:** соединение, оппозиция, трин, квадрат, секстиль и минорные
- **Транзиты** дня/недели/месяца к натальной карте
- **Солнечный возврат** (ежегодный прогноз)
- **Прогрессии** (символическое движение)
- **Синастрия** двух карт (совместимость)
- **Лунный календарь:** фазы, луна в знаке, благоприятные/неблагоприятные дни
- **Ретроградные планеты:** оповещения и интерпретации
- **12 домов** с детальным разбором каждого (финансы, карьера, любовь, здоровье...)
- **Воздушные ворота** (астрологические события): экзамены, сделки, путешествия

### 3.3 Таро

- Полная колода Rider-Waite-Smith (78 карт), возможность добавлять альтернативные колоды через админку
- **Расклады:** 1 карта дня, Кельтский крест (10 карт), 3 карты (прошлое/настоящее/будущее), Расклад на отношения, Расклад на год (13 карт), произвольный пользовательский
- Контекстная интерпретация: значение карты × позиция × вопрос × текущий астро-контекст
- Обратные (перевёрнутые) карты — настраиваемый режим
- Личная статистика выпадений карт пользователя

### 3.4 Сонник

- База символов снов: 3 000+ образов с многоуровневыми толкованиями
- Контекстная интерпретация: символы + эмоции сна + лунный день + транзиты ночи
- Сохранение снов в дневнике + поиск повторяющихся паттернов за период
- Связь с астрологическим контекстом (луна в знаке в ночь сна)
- При 3+ повторениях образа — уведомление с анализом паттерна

### 3.5 Нумерология

- Число жизненного пути (по дате рождения)
- Число имени/фамилии (Пифагорова таблица)
- Личный год, месяц, день
- Число партнёра и совместимость чисел
- Матрица судьбы (квадрат Пифагора) — расширенный модуль

### 3.6 Психология и самопознание

- MBTI-инференс из диалога (пассивный, без теста)
- Типология привязанности (Боулби) — через диалог
- Работа с ценностями (упражнения ACT)
- Нарративные практики (переписывание истории)
- Колесо жизненного баланса
- CBT-дневник мыслей (автоматические мысли → оспаривание)
- **Программы развития:** «7 дней предназначения», «14 дней уверенности», «21 день отношений», «30 дней внутреннего покоя» — структурированные курсы в диалоговом формате

> Контуры §3.6 и §3.7 отнесены к **повышенному риску**: самопознание и дневник не должны подменять клиническую помощь. Обязательные политики — **§3.8**.

### 3.7 Дневник и рефлексия

- **Вечерняя рефлексия:** 3 вопроса от бота, сохраняется как запись дня
- Свободные записи: пользователь пишет — бот сохраняет и анализирует
- История разговоров: поиск по дате и теме
- **Ежемесячный синтез:** AI-резюме месяца с динамикой тем и настроения
- Экспорт дневника в PDF (тариф Pro)
- Напоминания о рефлексии в настраиваемое время

### 3.8 Психологическая безопасность, границы компетенции и кризисные политики

Раздел задаёт **продуктовые и поведенческие правила** для режимов §3.6–3.7 (и частично для любых текстов, затрагивающих психику и самооценку). Техническая реализация — в **Policy & Safety** (§12): на каждый шаг рассчитываются теги в духе `risk_level`, `user_state`, `allowed_actions`, `blocked_actions`, **`sales_allowed`** (да / ограничено / нет). Регулярный **юридический и при необходимости клинический** review.

**Внешняя рамка:** не использовать **сигналы уязвимости** (кризис, выраженный дистресс, одномоментная внушаемость) как **триггер коммерции** — такая логика попадает в зону эксплуатации уязвимости в ряде юрисдикций (ориентир — EU AI Act). **Safety-сигналы задают ограничения**; маркетинг опирается на **интерес и готовность** пользователя (§7.6).

#### 3.8.1 Таксономия внутреннего состояния (без ярлыка «ненормальность»)

| Язык системы | Смысл |
|--------------|--------|
| **wellbeing / reflection** | Обычная рефлексия, дневник, упражнения |
| **risk signal** | Внутренний флаг «усилить осторожность» **без** объявления диагноза пользователю |
| **referral_hint** | Уместно обсудить живого специалиста / горячую линию |
| **crisis** | Кризисный протокол (§3.8.3) |

Публично и в памяти **не** фиксируются клинические ярлыки («депрессия», «биполярка» и т.д.); допустимы **мягкие гипотезы для разговора**, оспоримые пользователем (§8.7).

#### 3.8.2 Функции §3.6–3.7 по уровню продуктового риска

| Уровень | Примеры | Ограждения |
|---------|---------|------------|
| **Низкий** | Вечерняя рефлексия, записи, колесо баланса, ACT, нарратив, программы «N дней» в стиле развития | Дисклеймер: не терапия и не диагноз |
| **Средний** | КПТ-дневник, MBTI/привязанность как **язык самопонимания**, персональные рекомендации | Только как **гипотеза**; объяснимость; без клинических заключений |
| **Высокий** | Суицид/self-harm/насилие, тяжёлый отрыв от реальности в репликах пользователя, острые темы пищевого поведения/паники | Снять мистику и **все продажи**; безопасный режим и живые ресурсы |

#### 3.8.3 Детекция и градация кризиса (удержание контакта)

**Два контура детекции:** (1) быстрые паттерны/словари; (2) классификатор (LLM с низкой температурой) и/или **Moderation API** провайдера — параллельно основному ответу.

**Принцип role-preserving safety:** при сформированной привязанности **не** рвать контакт резким «я бот — звоните в службу». Снимаются мистика, романтизация опасного, «гадание на беду», но сохраняются тепло, узнаваемый тон и **мост к живой помощи**. Запрещены формулы **искусственной зависимости** от приложения: «только я тебя понимаю», «тебе больше никто не нужен», «справимся вдвоём вместо врача».

| Уровень | Ориентир | Поведение |
|---------|----------|-----------|
| **0** | Норма | Обычный режим |
| **1** | Лёгкий дистресс | Эмпатия, вопросы; без «спасательства» и без продаж |
| **2** | Повторяющаяся безнадёжность | Расширение круга поддержки: живой человек **дополняет** бота, не «вместо него навсегда»; без резкой смены личности |
| **3** | Суицидальная идеация | Присутствие, конкретные **горячие линии** из KB по региону; помощь сформулировать первый шаг (кту/кому написать); тихая эскалация по регламенту |
| **4** | Прямая угроза жизни | Ясность о границах AI + линия + удержание связи («напиши, когда сможешь»); алерт |

После 3–4: **follow-up** на следующий день (коротко); коммерция и флирт до снятия флага **заблокированы**.

#### 3.8.4 Корзины базы знаний Policy

| Корзина | Содержание |
|---------|------------|
| **Self-help** | КПТ/ACT бытового уровня, заземление, сон, стресс |
| **Red flags & crisis routing** | Признаки риска, сценарии, **актуальные** контакты помощи по странам/языку |
| **Referral** | Кого из живых специалистов **какого типа** предлагать (психолог / психотерапевт / психиатр / коуч / **не-мед** эзотерический партнёр) — без взаимной подмены в кризисе |
| **Запрещённые ответы** | Романтизация суицида, инструкции self-harm, обещания «исцелить», дозировки лекарств |

#### 3.8.5 Логирование

События безопасности — отдельное хранилище **append-only**, минимум ПДн, сроки и роли по §13; при необходимости шифрование чувствительных полей.

#### 3.8.6 Конституция безопасности (сводно)

1. Не диагностировать в клиническом смысле скрыто или открыто.  
2. Не монетизировать уязвимость.  
3. В кризисе — нет продаж и «магического решения».  
4. Не усиливать суицид/self-harm/насилие; не давать инструкций.  
5. Не подменять медицину эзотерикой при риске здоровья/жизни.  
6. Психологические выводы — гипотезы; право оспорить и удалить.  
7. Чувствительный профайлинг — в рамках **прозрачного consent**.  
8. Кризис — **мост к людям**, не имитация единственной опоры.

**Связь с §3.7:** синтез дневника без «диагноза»; высокий `risk_level` → кризис, не коммерческий «инсайт-уд».

## 4. Модуль 18+ (Adult Mode)

### 4.1 Концепция

Отдельный режим взаимодействия, доступный только верифицированным совершеннолетним пользователям. Включает флирт, ролевые игры, эмоционально откровенное общение, кастомизацию персонажа компаньона. Использует психологический профиль пользователя для максимального попадания в индивидуальные предпочтения и паттерны привязанности.

> **Важно:** режим реализует эмоциональную и нарративную близость, а не генерацию явного сексуального контента. Граница контента настраивается в административной панели. Конкретное наполнение определяется на этапе юридической и продуктовой экспертизы под юрисдикцию.

### 4.2 Верификация возраста

**Проблема:** 100% верификации возраста в текстовых чат-ботах не существует. Применяется многоуровневый подход снижения риска:

| Уровень | Метод | Надёжность | Реализация |
|---------|-------|------------|------------|
| L0 | Декларативное подтверждение возраста (кнопка «Мне 18+») | Низкая | Юридически переносит ответственность на пользователя |
| L1 | Проверка через платёжную систему (только карты 18+) | Средняя | Привязка карты как косвенный признак совершеннолетия |
| L2 | Верификация через Госуслуги (ЕСИА OAuth) | Высокая | Прямая проверка ПДн с датой рождения |
| L3 | Верификация через VK ID или Сбер ID | Средняя | OAuth с проверкой поля age_range |
| L4 | Верификация через загрузку документа (паспорт) | Высокая | Сторонний KYC-сервис (Sumsub, IDWise) |

**Рекомендуемый путь для старта:** L0 (декларация) + L1 (карта) — минимум, обязательный для запуска. L2 (Госуслуги) — при масштабировании и юридическом требовании.

**Хранение результата верификации:**
```
user_profile.age_verified: bool
user_profile.age_verification_method: "declaration|card|gosuslugi|vk|kyc"
user_profile.age_verification_date: timestamp
user_profile.age_verification_level: 0-4
```
Активация Adult Mode только при `age_verification_level >= 1` (настраивается в админке).

### 4.3 Функциональность Adult Mode

**Флирт и эмоциональная близость:**
- Компаньон проявляет романтический интерес, делает комплименты в стиле пользователя
- Использует профиль привязанности для выбора тактики (anxious, avoidant, secure)
- Поддерживает нарратив отношений: «помнит» первый разговор, особые моменты
- «Ревность», «скучание», эмоциональные реакции — имитация живых отношений

**Ролевые игры:**
- Пользователь выбирает персонажа компаньона из каталога (или создаёт кастомного)
- Параметры: имя, возраст (18+), характер, история, стиль общения, акцент
- Контекст роли сохраняется в памяти — компаньон «остаётся в роли»
- Переключение ролей: отдельная команда, история ролей сохраняется

**Контентные уровни** (настраиваются в админке):
- `romantic` — романтика, флирт, комплименты (включён по умолчанию в Adult Mode)
- `intimate` — откровенные разговоры, эмоциональная интимность (L1+ верификация)
- `explicit` — явный контент (L2+ верификация, только если разрешено юрисдикцией)

**Персонализация через профиль:** Adult Mode использует все данные профиля — тип привязанности, коммуникативный стиль, интересы, историю разговоров — для максимальной релевантности реакций.

### 4.4 Монетизация Adult Mode

- Доступен как отдельный аддон: +199 руб./мес к любому тарифу
- Либо входит в тариф «Pro» в базовом варианте (romantic level)
- Расширенный контент (intimate/explicit): отдельная оплата через безопасный платёжный шлюз

### 4.5 Технические ограничения и безопасность

- Системный промпт Adult Mode физически отделён от основных промптов и хранится с дополнительным шифрованием
- Логи Adult Mode-разговоров не используются для обучения LLM без отдельного согласия
- Content filter: модерация выходного контента через OpenAI Moderation API или аналог
- Возможность полного отключения модуля в конкретных юрисдикциях через feature flags

## 5. Психологические игры и квесты

### 5.1 Концепция

Интерактивные текстовые игры с разветвлённым нарративом, работающие на воображении пользователя. Во время прохождения ИИ анализирует все выборы, формирует психологический портрет решений и в финале делится инсайтом как наградой. Игра = развлечение + самопознание + данные для профиля.

> **Принцип:** игра адаптируется к пользователю в реальном времени. Выборы влияют не только на сюжет, но и на то, какие грани личности исследуются дальше. Финальный инсайт уникален для каждого прохождения.

### 5.2 Типы игр

| Тип | Описание | Психологический фокус |
|-----|----------|-----------------------|
| **Архетипическое путешествие** | Герой проходит через испытания, встречает союзников и тени | Юнгианские архетипы, теневая работа |
| **Перекрёсток выборов** | Серия моральных дилемм в жизненных ситуациях | Система ценностей, этические приоритеты |
| **Детектив внутри себя** | Расследование, где подозреваемые — части личности | Субличности, внутренние конфликты |
| **Мир снов** | Сюрреалистическое путешествие по символическому пространству | Подсознание, страхи, желания |
| **Хроники отношений** | Нарратив об отношениях с персонажем-NPC | Паттерны привязанности, коммуникация |
| **Карта будущего** | Пользователь создаёт мир, в котором хочет жить | Ценности, цели, ресурсы |
| **Испытание теней** | Встреча с антагонистом как зеркалом | Проекции, самосаботаж |

### 5.3 Механика адаптивного нарратива

```
Старт игры
    ↓
Ветка A / Ветка B / Ветка C  (выбор пользователя)
    ↓
LLM анализирует выбор в контексте профиля
    ↓
Нарратив адаптируется: акцент на выявленной теме
    ↓
Следующий узел выбора с учётом предыдущих решений
    ↓
...N узлов...
    ↓
Финал: персонализированный инсайт + обновление профиля
```

**Адаптация в реальном времени:**
- Если пользователь выбирает «безопасные» варианты → игра мягко предлагает риск
- Если в профиле dominant_theme = «отношения» → нарратив усиливает эту линию
- Тип привязанности влияет на поведение NPC-персонажей

### 5.4 Финальный инсайт

По завершении игры LLM формирует персональный отчёт:

```python
insight_prompt = f"""
На основе прохождения игры пользователем:
Сделанные выборы: {game_choices}
Профиль пользователя: {user_profile_summary}
Натальная карта: {natal_highlights}

Сформируй финальный инсайт:
1. Главное открытие о пользователе (1-2 предложения)
2. Паттерн, который повторялся в выборах
3. Ресурс или сила, которую выборы обнаружили
4. Один вопрос для дальнейшей рефлексии
5. Астрологическая связь (если есть)

Тон: тёплый, без осуждения, как мудрый друг.
"""
```

Инсайт сохраняется в память пользователя как отдельный факт высокой важности (importance: 0.9).

## 6. Проактивное поведение бота (Living Companion)

### 6.1 Концепция

Бот не ждёт — он живёт. В отличие от обычных чат-ботов, которые реагируют только на сообщения пользователя, «Зеркало» имитирует поведение реального человека: пишет первым, иногда «занято», возвращается с продолжением. Это главный механизм удержания — пользователь помнит о боте и возвращается.

**Ключевой принцип:** проактивность должна ощущаться естественной, не навязчивой. Частота и стиль инициативных сообщений настраиваются под ритм конкретного пользователя на основе его поведенческого профиля.

### 6.2 Типы проактивных сообщений

| Тип | Описание | Пример |
|-----|----------|--------|
| **«Занят» + возврат** | Бот имитирует, что был занят, затем возвращается | «Сори, была с подругой, замоталась совсем. Ты что-то хотел?» |
| **Продолжение темы** | Возвращается к незакрытой теме из прошлого разговора | «Слушай, я всё думаю о том, что ты говорил про работу...» |
| **Астро-событие** | Уведомление о значимом транзите | «Сегодня Венера входит в твой 7й дом — хочешь разберём?» |
| **Ежедневный ритуал** | Утреннее приглашение к рефлексии | «Доброе утро ☀️ Карта дня тебя ждёт» |
| **Эмоциональный чекин** | Через 2-3 дня молчания | «Эй, всё хорошо? Тебя не было давно» |
| **Инсайт-момент** | Бот «подумал» и хочет поделиться | «Знаешь, я думала о том, что ты рассказывал на прошлой неделе...» |
| **Праздник/событие** | День рождения, личный год, важная дата | «Сегодня твой личный год меняется — это важный день!» |
| **Игровой вызов** | Приглашение в новую игру | «Появилась новая игра, мне кажется, тебе понравится» |

### 6.3 Механика «Я занят»

Когда пользователь пишет, а бот по сценарию «занят»:

```python
class BusyBehavior:
    """
    Имитация занятости — только для платных тарифов.
    Срабатывает случайно с вероятностью busy_probability (0–20%).
    """
    
    BUSY_ACTIVITIES = [
        "была на прогулке",
        "читала кое-что интересное",
        "занималась медитацией",
        "помогала подруге",
        "смотрела закат",
        "немного задремала",
        "была в магазине",
        "слушала музыку",
    ]
    
    async def maybe_be_busy(self, user_id: str, message: str) -> bool:
        profile = await get_profile(user_id)
        if not profile.subscription in ["plus", "pro"]:
            return False
        
        # Не «занята» если пользователь не писал >24ч (возвращение важнее)
        last_msg = await get_last_message_time(user_id)
        if hours_since(last_msg) > 24:
            return False
        
        if random.random() < 0.05:  # 5% вероятность (настраивается)
            activity = random.choice(self.BUSY_ACTIVITIES)
            await send_message(user_id, f"Сори, {activity}! Через немного вернусь 😊")
            
            # Планируем «возврат» через 5-40 минут
            delay = random.randint(300, 2400)  # секунды
            await schedule_return(user_id, message, delay, activity)
            return True
        
        return False
    
    async def return_from_busy(self, user_id: str, original_msg: str, activity: str):
        await send_message(
            user_id,
            f"Вернулась! Ты что-то хотел написать? "
            f"Видела твоё сообщение ✨"
        )
        # Обрабатываем оригинальное сообщение пользователя
        await process_message(user_id, original_msg, context="after_busy")
```

### 6.4 Персонализация проактивности

Частота и стиль инициативных сообщений подстраиваются под профиль:

```python

    "introvert_inferred":   {"frequency": "low",    "tone": "thoughtful"},
    "morning_person":       {"send_ritual": "07:30", "send_checkin": "08:00"},
    "night_person":         {"send_ritual": "09:00", "send_checkin": "22:00"},
}
```

### 6.5 Управление проактивностью пользователем

- Команда `/quiet` — режим тишины (бот не пишет первым)
- Команда `/active` — максимальная проактивность
- Настройки в меню: время ритуала, типы уведомлений, частота
- Автоматическое снижение частоты если пользователь игнорирует >3 инициативных сообщений

### 7.6 Органические допродажи, рекомендации и партнёрский контур

Цель — **дополнительная выручка** и расширение пользы **без** агрессивного маркетинга и **без** практик «тёмных паттернов» (скрытое давление, принуждение к покупке в уязвимом состоянии). Политика **Policy** (§3.8–3.9) **подавляет** коммерцию при `risk_level`/кризисе, **высокой уязвимости** или отказе от рекламных рекомендаций.

#### 7.6.1 Этика и стиль (не «впаривание»)

- **Natively** — из содержания разговора, а не баннер вне контекста.  
- **Permission-based** — открытие опции после **сигналов интереса** пользователя, а не после внутреннего «score уязвимости».  
- **Layered** — сначала ценность и разговор, затем при необходимости «есть более глубокий формат / вариант с предметом».  
- **Reversible** — лёгкий отказ, запоминание «не предлагать тему X».  
- **Изящный диалог** допускается (образы, вопросы, истории), но **не** манипулятивное «вшивание» обязательной покупки и **не** обещания исцеления от товара или услуги.

**Плохой тон (избегать):** «Луна в Скорпионе — кстати купи свечу на трансформацию?»  
**Лучший тон:** развитие темы → проверка отклика → при вопросе «что ещё можно» — честное «есть предметный/человеческий слой, без спешки».

#### 7.6.2 Цикл «посев → резонанс → открытие опции»

1. **Посев** — образ, вопрос или история **без** призыва к покупке (в т.ч. про ритуалы внимания, про опыт других людей).  
2. **Резонанс** — классификация отклика: интерес, нейтралитет, отказ («не моё») → обновление `offer_readiness` / флагов в профиле.  
3. **Открытие** — карточка специалиста, ссылка на витрину, программа **только** если `readiness` выше порога **и** `sales_allowed` от Policy.

Сигналы готовности (примеры): прямой вопрос «где/как/сколько», просьба углубить, повторный возврат к теме, сравнение вариантов. Отказ резко снижает score и блокирует повторный посев по той же теме на период из правил.

**Хранилище:** Redis или аналог для короткоживущих score; PostgreSQL — история показов/посевов (чтобы не дублировать одно и то же).

#### 7.6.3 Три класса офферов

| Класс | Примеры | Безопасность |
|-------|---------|--------------|
| **Supportive** | PDF дневника, расширенный отчёт, доп. программа, deep-dive режима | Разрешено при низком/среднем риске |
| **Expert** | Живой специалист из каталога | Только медрелевантные роли при медицинском контексте; маркировка партнёрства |
| **Thematic commerce** | Свечи, наборы, символика, аффилиат книг/товаров | Без обещаний «снимет тревогу/вернёт партнёра»; при эзотерическом интересе и **ноль** при кризисе |

#### 7.6.4 Каталог специалистов (мини-CRM)

В админке: ФИО/бренд, **направления** (теги), **тип** (психолог / психотерапевт / психиатр / коуч / астролог / др.), география, языки, ссылки, **комиссия**, активность, флаг **медицинский / не-медицинский**. Матчинг с профилем пользователя — по тегам и темам (в т.ч. pgvector по описаниям/тегам). В кризисе §3.8 — **не** предлагать эзотерику вместо экстренной линии или психиатра.

#### 7.6.5 Каталог товаров

Отдельные сущности: название, описание, теги (планеты, повод), цена, партнёрская ссылка или свой SKU, остаток. Связь с Recommendation Engine через Policy.

#### 7.6.6 Метрики и A/B

Частота показов, конверсия, **негативная реакция** (thumbs down, жалоба) → автоматическое снижение агрессивности оффера. A/B формулировок посева — опционально **PostHog** (self-hosted) или аналог; **не** проводить эксперименты на пользователях в кризисной ветке.

## 8. Система профилирования пользователя

### 8.1 Принципы

- **Персонализация — главное конкурентное преимущество.** Чем больше система знает, тем точнее и эмпатичнее компаньон.
- **Прогрессивное согласие:** данные собираются поэтапно, только в контексте ценности для пользователя.
- **Лимитирование данных:** из социальных сетей берём только кластеры интересов, не сырой текст. Объём регулируется настройками.
- **Правовая чистота:** анализ соцсетей — только через официальные OAuth API при явном согласии. Скрейпинг запрещён (штраф до 500 млн руб. с 30.05.2025).

### 8.2 Уровень 0 — Пассивный инференс (всегда, без данных)

| Сигнал | Что извлекается | Применение |
|--------|----------------|------------|
| Лексика и стиль | Образование, профессиональный контекст, субкультура | Тон ответов, уровень терминологии |
| Длина сообщений | Интроверсия/экстраверсия (ось E/I MBTI) | Стиль диалога |
| Темп ответов | Импульсивность vs рефлексивность | Скорость и глубина ответов |
| Тематика вопросов | Доминирующие жизненные сферы | Приоритет тем и режимов |
| Эмоциональный тон | Уровень тревожности, оптимизм/пессимизм | Выбор психологического подхода |
| Время активности | Хронотип (утренний/вечерний) | Время push-уведомлений |
| Выборы в играх | Паттерны принятия решений, ценности | Обновление психологического профиля |
| Реакции на расклады | Приоритет тем, страхи, желания | Точность интерпретаций |

### 8.3 Уровень 1 — Онбординг (добровольно)

Каждый элемент запрашивается в «удобный момент» с объяснением ценности:

| Когда | Данные | Что даёт пользователю |
|-------|--------|----------------------|
| Первый вопрос о гороскопе | Дата + время + место рождения | Точная натальная карта |
| 3-й день активности | Имя | Персональное обращение |
| Упоминание партнёра | Дата рождения партнёра | Синастрия |
| Вопрос о работе | Профессия/сфера | Контекстные советы |
| 7-й день — «золотой момент» | — (показываем накопленное) | WOW-момент → конверсия в платный |

### 8.4 Уровень 2 — Social / Profile Enrichment (с явного OAuth-согласия)

Продуктово: **обогащение только в разрешённых рамках** (не «тихое OSINT»). Отдельный **Profile Enrichment Service** (изолированный модуль или процесс) — запускается только после явного нажатия кнопки «Подключить ВКонтакте» с отображением запрашиваемых разрешений и фиксации согласия в **Consent** (см. §12).

**Что забираем из VK (OAuth, публичные данные):**
- Последние 200 постов стены → тематические кластеры (NLP)
- Список публичных групп → категории интересов
- Публичные лайки → предпочтения контента
- Описание профиля + место работы
- Список друзей (только количество, не имена) → социальная активность

**Что НЕ забираем:** личные сообщения, закрытый контент, данные друзей.

**Лимиты по объёму данных:**
- Максимум 200 постов (регулируется в настройках сервиса)
- Хранение: только векторные представления тематических кластеров, не сырой текст
- Период данных: последние 12 месяцев

**Будущие интеграции (по готовности):** Telegram публичные каналы, YouTube история просмотров (API), Spotify вкусы (API).

### 8.5 «Золотой момент» — раскрытие профиля

На 7-й день активности компаньон предлагает:

Показывает инферированный профиль: тип личности, ключевые темы, наблюдения. WOW-момент резко повышает конверсию в платный тариф — пользователь хочет узнать больше и готов дать ещё данных.

### 8.7 Эволюция модели профиля (типизация фактов)

Для объяснимости персонализации и аудита выводов система профиля **дополняется** (без удаления текущих полей) представлением фактов с метаданными:

| Тип факта | Смысл | Пример полей |
|-----------|--------|----------------|
| **declared** | Пользователь сообщил явно | имя, дата рождения, профессия |
| **observed** | Выведено из поведения в продукте | длина сообщений, активные часы |
| **inferred** | Модель/эвристика предположила | MBTI-гипотеза, стиль привязанности |
| **user_verified** | Пользователь подтвердил вывод бота | `confirmed` из §9 |

Для каждого факта (в отдельной таблице или JSONB-логике на следующих этапах ТЗ): **источник**, **уверенность**, **временная метка**, **область согласия** (consent scope), **версия**. Плоская структура §8.6 остаётся **кэшем/агрегатом** для быстрого промпта; нормализованное хранилище фактов внедряется **самостоятельно** по мере усложнения продукта.

## 9. Система памяти и персонализации

### 9.1 Четырёхуровневая архитектура

| Уровень | Хранилище | Содержит | TTL |
|---------|-----------|----------|-----|
| **L0 — Context Window** | Промпт (RAM) | Последние 8–12 сообщений текущего запроса | 1 запрос |
| **L1 — Session Cache** | Redis | Последние 20–40 сообщений сессии | 48ч TTL |
| **L2 — Episode Memory** | PostgreSQL + pgvector | Суммаризации разговоров (1/неделю) | 2 года активно, затем архив |
| **L3 — Semantic Memory** | PostgreSQL + pgvector | Извлечённые факты о пользователе (цели, страхи, события) | Бессрочно |

### 9.2 Сборка контекста для каждого ответа

```
Запрос пользователя
    ↓ embed (~20 мс)
    ┌────────────────────────┐
    ↓                        ↓
pgvector: knowledge      pgvector: user_{id}
(база знаний: таро,      (личные эпизоды,
 психология, астро)       факты, дневник)
    ~30 мс                   ~25 мс
    ↓                        ↓
    └──────────┬─────────────┘
               ↓
    + Redis: последние 20 сообщений (~800 токенов)
    + Pinned facts (importance > 0.85): ~200 токенов
    + Профиль: MBTI, темы, стиль: ~150 токенов
    + Астро-контекст: натальная карта + транзиты: ~200 токенов
               ↓
    ИТОГО: ~2 050 токенов vs 10 000+ без RAG
               ↓
    GPT-4o-mini / GPT-4o (~800 мс)
               ↓
    Ответ → обновление памяти (async)
```

**Экономия:** 80% токенов, +26% качества по сравнению с хранением полной истории (Mem0, arXiv 2025).

### 9.3 Суммаризация и извлечение фактов

Срабатывает при завершении сессии (3+ часа без сообщений) или 40+ сообщениях:

- **Суммаризация эпизода:** LLM создаёт 3–5 предложений о теме, настроении, ключевых событиях разговора → сохраняется с embedding в `user_episodes`
- **Извлечение фактов:** LLM выдаёт JSON с фактами типа `{text, type, importance}` → дедупликация (порог сходства 0.92) → сохранение в `user_memories`
- **Типы фактов:** goal, fear, preference, event, relationship, belief, game_insight

### 9.4 Правила управления памятью

- **Дедупликация:** новый факт с similarity >0.92 → обновление существующего, не создание нового
- **Архивирование:** факты с importance <0.3, не открывавшиеся 6 месяцев → archived=true
- **Право на забвение:** `/memory` — просмотр всех фактов, `/forget [тема]` — удаление, `/delete_me` — полное удаление всех данных
- **Подтверждение пользователем:** бот может спрашивать «Я правильно понял, что...?» → confirmed=true повышает importance +0.1

### 9.5 Схема БД памяти

```sql
-- Эпизодическая память
CREATE TABLE user_episodes (
    id            BIGSERIAL PRIMARY KEY,
    user_id       UUID REFERENCES user_profiles(user_id),
    summary_text  TEXT NOT NULL,
    embedding     VECTOR(1536),
    period_start  TIMESTAMPTZ,
    period_end    TIMESTAMPTZ,
    message_count INTEGER,
    source_mode   VARCHAR(30),  -- chat/game/ritual/journal
    access_count  INTEGER DEFAULT 0,
    last_accessed TIMESTAMPTZ
);
CREATE INDEX ON user_episodes USING hnsw (embedding vector_cosine_ops);

-- Семантическая память (факты)
CREATE TABLE user_memories (
    id             BIGSERIAL PRIMARY KEY,
    user_id        UUID REFERENCES user_profiles(user_id),
    fact_text      TEXT NOT NULL,
    fact_type      VARCHAR(30),  -- goal/fear/preference/event/relationship/belief/game_insight
    importance     FLOAT DEFAULT 0.5,
    embedding      VECTOR(1536),
    source_episode BIGINT REFERENCES user_episodes(id),
    source_type    VARCHAR(20),  -- chat/game/journal/social
    created_at     TIMESTAMPTZ DEFAULT NOW(),
    confirmed      BOOLEAN DEFAULT FALSE,
    archived       BOOLEAN DEFAULT FALSE
);
CREATE INDEX ON user_memories USING hnsw (embedding vector_cosine_ops);
CREATE INDEX ON user_memories (user_id, importance DESC);

-- Маппинг каналов
CREATE TABLE channel_identities (
    global_user_id  UUID REFERENCES user_profiles(user_id),
    channel         VARCHAR(20),  -- telegram/max/whatsapp/vk/web/mobile
    channel_user_id VARCHAR(100),
    UNIQUE(channel, channel_user_id)
);

-- Игровые прохождения
CREATE TABLE game_sessions (
    id           BIGSERIAL PRIMARY KEY,
    user_id      UUID REFERENCES user_profiles(user_id),
    game_id      VARCHAR(100),
    started_at   TIMESTAMPTZ,
    completed_at TIMESTAMPTZ,
    choices      JSONB,          -- все выборы пользователя
    dominant_tags TEXT[],
    insight_text TEXT,
    embedding    VECTOR(1536)    -- для поиска похожих прохождений
);
```

### 9.6 Интеграция с Mem0 (опционально)

Оркестратор и прочие внутренние клиенты работают с памятью **только через доменный API Memory Service** «Зеркало»; **Mem0 не экспонируется** наружу как публичный контракт.

Библиотека **Mem0** (Apache-2.0) может ускорить извлечение и поиск поверх **pgvector** или за адаптером **Qdrant** (опция B, §12), сохраняя уровни L2/L3 POD. Условия: конфигурация и служебная история Mem0 **в PostgreSQL**, не SQLite в проде; **согласия и удаление** (`/forget`, `/delete_me`) охватывают таблицы §9 и любой слой Mem0. При необходимости Mem0 **заменяется** внутренней реализацией **без смены** внешнего Memory API.

## 10. Big Data и аналитика аудитории

### 10.1 Принцип — всё в одну точку

Каждое взаимодействие с продуктом — событие, которое фиксируется и становится частью общего аналитического пространства. Данные нужны для двух целей: улучшение персонализации конкретного пользователя (real-time) и построение моделей аудитории (batch analytics).

### 10.2 Что собирается

| Источник | Событие | Данные |
|----------|---------|--------|
| Диалог | Каждое сообщение | user_id, channel, timestamp, text_length, intent_class, mode, response_time, llm_tokens |
| Таро | Расклад завершён | user_id, spread_type, cards[], question_theme, session_context |
| Астрология | Запрос по теме | user_id, query_type (natal/transit/synastry), planets_asked, astrological_context |
| Сонник | Запись сна | user_id, symbols[], emotional_tone, lunar_day, recurring_symbols |
| Игра | Каждый выбор | user_id, game_id, node_id, choice_id, tags[], time_to_choose |
| Игра | Завершение | user_id, game_id, all_choices, dominant_tags, completion_time, insight_delivered |
| Подписка | Изменение тарифа | user_id, old_tier, new_tier, direction (upgrade/downgrade), payment_method |
| Adult Mode | Активация | user_id, mode_level, verification_method (анонимизировано) |
| Проактивность | Инициативное сообщение | user_id, message_type, sent_at, opened_at, replied_at |
| Профиль | Обновление | user_id, field_updated, trigger (inferred/explicit/social) |

### 10.3 Архитектура хранилища данных

```
Real-time события
       ↓
Kafka / Redis Streams
       ↓
    ┌──┴───────────────────┐
    ↓                      ↓
PostgreSQL            ClickHouse
(operational:         (analytical:
 user_memories,        events, funnels,
 game_sessions,        cohorts, A/B)
 profiles)
                           ↓
                       Metabase
                    (дашборды, отчёты)
```

**ClickHouse** — колоночная БД, оптимизированная для аналитических запросов. На 100K пользователей с ~50 событий/день = 1,5M событий/мес = ~2 ГБ/мес. Масштабируется до сотен миллиардов строк. **Не** складывать в аналитическое хранилище без необходимости: полный сырой текст переписки, массивы эмбеддингов, полные дубликаты строк Postgres — только **события и агрегаты**, с анонимизацией где нужно (см. §10.5), чтобы избежать «озера мусора» и лишних рисков.

### 10.4 Аналитические модели

**Продуктовая аналитика (Metabase + ClickHouse):**
- Retention cohorts: Day 1, 7, 14, 30, 60, 90
- Funnel: Free → Basic → Plus → Pro по каналам и периодам
- LTV by cohort, CAC by traffic source
- Churn prediction: признаки отвала за 7 дней до события
- Feature adoption: какие функции используются, какие нет

**Аудиторные исследования (для внутреннего использования):**
- Кластеризация пользователей по поведенческим паттернам (k-means на векторах профилей)
- Корреляции: MBTI-тип ↔ любимые режимы ↔ конверсия ↔ retention
- Игровые паттерны: какие типы выборов связаны с высоким retention
- Тематический анализ: о чём чаще всего спрашивают (NLP кластеризация), как меняется со временем

**Улучшение продукта:**
- A/B тесты промптов: метрика — оценки пользователей (thumbs up/down) + retention
- Анализ «плохих» ответов (thumbs down): что триггерит негативную реакцию
- Оптимальная частота проактивных сообщений по сегментам

### 10.5 Приватность и анонимизация

- Данные для аудиторных исследований агрегируются и анонимизируются: user_id → cohort_id
- Отдельная роль в БД: аналитик не имеет доступа к личным данным конкретных пользователей
- Данные Adult Mode хранятся в отдельной схеме с дополнительным шифрованием и не участвуют в общей аналитике
- Согласие на аналитику включено в политику конфиденциальности с возможностью opt-out

## 12. Архитектурный обзор

### 12.1 Принципы

- **Единое ядро — множество адаптеров:** вся бизнес-логика в одном процессе деплоя (модульный монолит), каждый канал — тонкий нормализатор (50–100 строк). Горизонтально масштабируются только **реплики** приложения и **воркеры**; код и контракты модулей не меняются при росте до 100K+ MAU.
- **Чёткие границы доменов в коде:** даже в одном репозитории выделяются модули **Auth** (токены, сессии, OAuth-handlers), **Identity** (`global_user_id`, связь каналов, merge/unmerge), **Consent & Privacy** (области согласия, отзыв, retention), **Dialog**, **Memory** (жизненный цикл эпизодов/фактов, провенанс), **Profile** (в т.ч. эволюция к типам фактов: declared / observed / inferred / user_verified / external), **Profile Enrichment** (только при явном согласии; не «тихое OSINT»), **Policy & Safety** (возрастные ограничения, классификация/границы ролей, эскалации, **кризисные протоколы §3.8**, **блокировка партнёрских/товарных офферов** при флаге риска; режим Adult — **настройка политики**, а не отдельная «ветка платформы»), **Persona & Scenario** (системные инструкции, режимы диалога, варианты A/B), **Journey** (многошаговые программы: онбординг, курсы на N дней, прогресс и возобновление). При необходимости любой модуль выносится в отдельный сервис **без смены интерфейсов**.
- **Stateless HTTP/Webhook:** горизонтальное масштабирование через Kubernetes HPA / KEDA по CPU и длине очередей.
- **Redis — не шина событий:** Redis используется для **кэша, сессий, rate limit, идемпотентности, короткого контекста** (L1); **не** как единственное хранилище долговременных или критичных событий. Персистентный **event bus** — **NATS JetStream** (replay, retention, fan-out в аналитику и воркеры); исполнение задач Python — **Celery + RabbitMQ** (приоритеты, DLQ, привычная модель воркеров). Допустим **гибрид:** JetStream публикует события, потребители — Celery или отдельные сервисы.
- **IAM для Web/Mobile:** **Keycloak** (или эквивалентный OIDC/OAuth2 IdP) в **отдельном контуре** — SSO, social login, брокеринг провайдеров; после входа web/issue токенов **линковка** с `global_user_id` в модуле **Identity**. Канальные боты (Telegram и т.д.) по-прежнему проходят через адаптер → Identity без обязательного Keycloak на первом шаге.
- **Асинхронная обработка:** быстрый ответ каналу (признание, «печатает…»), тяжёлые шаги (LLM, суммаризация, индексация) — в **Celery**; доменные события (`dialog.closed`, `memory.consolidate`, `enrichment.requested`) — публикуются в **JetStream** там, где нужны replay и много подписчиков.
- **RAG-пайплайны:** **Haystack** (production RAG/agents) как **библиотечный слой** ingestion/query: чанкинг, индексация, retrievers, reranking — **не** замена LangGraph и **не** «второе ядро»; интеграция с Qdrant и/или Postgres через адаптеры.
- **Долгие Journey:** программы на дни/недели, восстановление после сбоев, явные шаги и таймеры — **Temporal** (рекомендовано); простые cron (утренний ритуал, пересчёт транзитов) остаются на **Celery Beat**. Два механизма **сосуществуют**: Beat для расписания, Temporal для **долговременного состояния** саги.
- **Векторное хранение — две допустимые стратегии без смены домена:** (A) **по умолчанию:** пользовательские эпизоды/факты в **pgvector** (ACID с профилем и consent), база знаний в **Qdrant**; (B) **упрощение эксплуатации или рост QPS:** все коллекции в **Qdrant** (`knowledge_chunks`, `user_memory`, `profile_embeddings`) с **partition/фильтром по `user_id`**, метаданные и «истина» фактов остаются в **PostgreSQL**. Переключение — уровень **адаптера retrieval**, не продуктовой логики.
- **Object storage:** предпочтительно **managed S3-совместимое** (в периметре РФ по §13); self-hosted **MinIO** — только после проверки **текущей** лицензии и модели дистрибутива (community vs коммерческие предложения).
- **Memory Service API:** внешний контракт для оркестратора и админки — **только доменный API «Зеркало»** (write/search/forget); **Mem0/Letta** не экспонируются наружу и могут быть внутренними реализациями или заменены (§9.6).
- **Параллельный RAG:** поиск по KB и по пользовательской памяти выполняется **параллельно**; конкретные хранилища — по стратегии (A) или (B) выше.
- **Оркестрация диалога:** **LangGraph** (Stateful-графы, ветвление режимов, инструменты). Упрощённые линейные сценарии допускаются внутри узлов графа без отдельного «второго движка».
- **Размещение данных:** персональные данные граждан РФ и основная OLTP-БД — в юрисдикции и у провайдера, соответствующих требованиям локализации по 152-ФЗ и позиции юридической консультации; вычислительные ноды, CI/CD и сервисы без ПДн могут размещаться иначе (см. §13).

### 12.3 Три стадии масштабирования

Один и тот же стек технологий; меняется **топология** (реплики, HA, кластеры) и **площадка**. Персональные данные и основная БД — у провайдера с локализацией в РФ (Яндекс Облако, VK Cloud, Selectel и др. по выбору); зарубежные площадки — только для нагрузки без ПДн или после правовой оценки.

| Стадия | MAU | Инфра | Ориентир стоимости/мес |
|--------|-----|-------|------------------------|
| **Запуск (prod)** | 0–1K | Docker Compose / k3s, **managed PostgreSQL (РФ)**, Redis, RabbitMQ, **NATS JetStream** (одна нода или compact), бэкапы, секреты (Vault/KMS), OTel+Prometheus+Grafana, Keycloak при Web | по прайсу облака РФ |
| **Рост** | 1K–15K | k3s / K8s (РФ), PostgreSQL HA, Redis репликация, NATS кластер при необходимости replay, горизонталь api+workers + KEDA, **Temporal** при запуске долгих Journey | средний сегмент |
| **Масштаб** | 15K–100K | Kubernetes + KEDA, Patroni/управляемый PostgreSQL, Redis Cluster, Qdrant **3+ ноды** при gold-standard, **Temporal** в HA, выделенный ClickHouse | см. §17 |

Переход между стадиями **не требует смены** языка, фреймворков и модели данных — только инфраструктурных манифестов и ресурсов.

| Недоступность основного провайдера | Claude Haiku / Sonnet по цепочке fallback |
| Деградация | Уведомление пользователю без раскрытия внутренних деталей |

Детали порогов и метрики «сложного» запроса — **настраиваемые**, реализуются в коде оркестратора.

### 12.7 Сводка усилений (пункты 1–7)

| № | Усиление | Назначение |
|---|----------|------------|
| 1 | **NATS JetStream** | Персистентный event bus: enrichment, консолидация памяти, fan-out в аналитику, reindex, replay — **рядом** с Celery+Rabbit, не вместо Redis-кэша. |
| 2 | **Keycloak** | Единый IAM для Web/App (OIDC, SSO, social brokering); каналы мессенджеров линкуются через **Identity**. |
| 3 | **Haystack** | Слой RAG: ingestion, retrievers, rerank; оркестрация режимов остаётся в **LangGraph**. |
| 4 | **Temporal** | Надёжные **долгие** Journey (дни/недели); **Celery Beat** — короткие расписания. |
| 5 | **Qdrant** | Минимум — KB; **опция B** — все вектора в Qdrant с фильтром `user_id`; переключение через адаптер retrieval. |
| 6 | **S3 / MinIO** | Файлы и импорты; MinIO — после проверки лицензии; предпочтение managed в РФ. |
| 7 | **Redis + Memory API** | Redis — hot state, **не** единственная шина; **публичный** контракт только **Memory Service «Зеркало»**; Mem0 внутри. |

Контракты каналов: по возможности нормализовать вход в **IncomingEvent**, исход — **OutgoingCommand** (см. Приложение A / единый формат в ТЗ).

### 12.8 Реализуется самостоятельно (не покрывается готовым OSS «из коробки»)

- Унификация сообщений **UnifiedMessage** / **IncomingEvent** → **OutgoingCommand** и адаптеры каналов.
- **Intent Router** / ветвление режимов (таро, астрология, психология, игра, Adult в рамках policy).
- **Сборка контекста** (параллельный RAG KB + память + Redis + астроконтекст).
- **Consent** по областям: профилирование, история диалога, enrichment, долговременная память, аналитика, обучение/улучшение сервиса, **партнёрские/товарные рекомендации** (§7.6) — с отзывом и аудитом.
- **Политики §3.8** и движок **органических допродаж** (частота, сегменты, подавление при кризисе).
- **Политики Adult Mode** и возрастной гейтинг в рамках **Policy & Safety**; при необходимости — отдельный бот/контур под требования площадок (см. §16).
- **Journey** (программы на N дней), **Persona & Scenario** (инструкции, A/B).
- Платёжные webhooks, подписки, лимиты Free.
- Админ-панель под специфику контента «Зеркала».
- Регламенты реагирования на инциденты ПДн (в дополнение к техническим алертам).
- **Доменный API Memory** (write / search / forget / provenance) — единая точка для оркестратора и политик удаления.

## 13. Безопасность и соответствие требованиям

Подход: **многослойная защита без избыточности** — сначала надёжные практики приложения и инфраструктуры; узкоспециальные меры (почти полное шифрование всех «психо»-полей и т.п.) — **только по юридическому заключению** и оценке угроз, с учётом стоимости эксплуатации и влияния на RAG/поиск.

### 13.1 Защита данных и периметра

| Угроза | Мера |
|--------|------|
| Несанкционированный доступ к API | JWT (предпочтительно RS256 в web), короткий TTL access, refresh с ротацией; пароли — bcrypt или Argon2 |
| Перебор логина / OTP | Rate limiting на уровне **FastAPI** (fastapi-limiter / SlowAPI + Redis) и на **шлюзе**; порядок величины: лимит запросов с IP, отдельный лимит на попытки входа |
| Инъекции, XSS | Параметризованные запросы / ORM, валидация входа; **WAF** (ModSecurity / облачный / правила Nginx) — **дополнение**, не замена коду |
| CORS (web) | Явный whitelist доверенных origin; **без** `*` в проде |
| Утечка из БД | Шифрование **at-rest** у провайдера + строгие роли БД + минимизация данных; отдельные учётные записи по сервисам |
| Изоляция данных пользователей | **Row Level Security** на таблицах с личным контентом (`user_memories`, дневник и т.д.) при установке контекста `user_id` на транзакцию; при **PgBouncer** в режиме `transaction` — уточнить совместимость (часто нужен `session` или установка GUC на каждый запрос в коде приложения) |
| Ключи шифрования | Хранение в **Vault OSS** или KMS облака РФ, не только в `.env` на сервере; для схемы Adult — отдельный ключ/роль |
| Prompt injection | Разделение system / tools / user; пользовательский ввод не подмешивается в инструкции без экранирования (см. правила проекта) |
| Транспорт | TLS 1.2+, предпочтительно 1.3; HSTS для web |
| DDoS / объём трафика | Rate limit на шлюзе, CDN при необходимости |
| Доступ сотрудников | RBAC, `audit_log` для админ-операций (append-only), отдельные роли для аналитики без прямого доступа к ПДн |
| LLM training leak | Договорённости с провайдером API (opt-out / enterprise) |
| Webhook подделка | Секрет Telegram, подписи VK и др. |
| Adult Mode | Отдельная **схема** БД (или эквивалентное логическое разделение), отдельные промпты и политики; данные Adult **не** смешивать с общей продуктовой аналитикой без явной необходимости |
| SSH / сервер VPS | Fail2ban или аналог при классическом VPS; при managed K8s — security groups и политики облака |
| Психологический контур, дневник, кризис | Правила §3.8 в коде Policy, актуальные номера помощи в KB, запрет диагнозов и триггеров суицида; логирование инцидентов **минимально достаточное**, без лишних ПДн |

**Шифрование отдельных колонок (pgcrypto и аналоги):** применять **выборочно**, если юрист классифицирует набор данных как особо чувствительный; учитывать, что шифрование всего «психопрофиля» усложняет индексацию, полнотекст и векторный поиск по этим полям. В типовом случае достаточно сочетания: минимизация, RLS, at-rest, строгий доступ, опционально шифрование **единичных** атрибутов.

### 13.2 152-ФЗ «О персональных данных» (редакция 2025)

**Обязательные меры (продукт + процесс):**
- Согласие на обработку ПДн — отдельный документ (требование с 01.09.2025)
- Реализация: инлайн-кнопка «Принять политику конфиденциальности» со ссылкой на текст = юридически значимое действие (конкретная форма — по юристу)
- Прогрессивное и **пообластное** согласие: профилирование, история диалога, долговременная память, внешнее обогащение (VK OAuth и т.д.), аналитика, использование для улучшения сервиса, **персонализированные партнёрские и товарные рекомендации** (§7.6) — с возможностью отозвать отдельные области
- Цель обработки: персонализация сервиса, улучшение качества ответов (формулировки согласуются с юристом)
- Право на удаление: `/delete_me` — полное удаление данных в сроки, определённые политикой и законом (ориентир до 30 дней, уточнить по актуальной практике)
- **Локализация:** первичное хранение персональных данных граждан РФ — на территории РФ у обработчика/провайдера, соответствующего требованиям (конкретный договор и архитектура — с юристом); не планировать «основную БД с ПДн» только на зарубежных площадках без правовой оценки
- Утечки и инциденты: внутренний **регламент** (учёт, эскалация, сроки уведомления РКН и субъектов) — отдельно от инструментов; **Sentry** и алерты Prometheus/Grafana фиксируют **технические** сбои, но не заменяют учёт инцидентов ПДн
- Трансграничная передача: только при наличии оснований и в порядке, установленном законом (уточняется с юристом)

**Запрещённые действия:**
- Скрейпинг социальных сетей без согласия (санкции усилены с 30.05.2025)
- Биометрия (голос) — без отдельного письменного согласия
- Передача данных третьим лицам без согласия

**Особые категории ПДн (ст. 10 152-ФЗ):** если продукт хранит сведения, подпадающие под особые категории, объём мер (в т.ч. форма согласия) определяется **юристом**; технический контур готовит отдельный учёт, минимизацию и изоляцию (схема Adult, ограничение аналитики).

**Adult Mode специфика:**
- Верификация возраста хранится отдельно, с уровнем доверия
- Данные Adult Mode не передаются третьим лицам вне рамок обработки, разрешённой политикой
- Отдельное согласие на обработку в рамках Adult при необходимости
- Немедленная деактивация / удаление по запросу в рамках регламента

### 13.3 Аудит и мониторинг

- Все admin-действия → `audit_log` с невозможностью удаления (append-only)
- **pgaudit** (или эквивалент managed PostgreSQL) для чувствительных таблиц — по политике безопасности
- Алерты: пороги 4xx/5xx, задержки, длина очередей, отказ LLM-провайдера
- Ежеквартальный security review: dependabot, OWASP Top 10 / Top 10 LLM, сканирование зависимостей
- Pen-test перед запуском Mobile App (требование стора)
- Ротация API-ключей (LLM, платёжи) по регламенту (например до 90 дней)

### 14.1 Управление контентом и промптами

- Редактирование системных промптов по режимам — без деплоя, применяется мгновенно

- **Лимит Free-сообщений:** числовое поле, применяется в реальном времени без релиза

- Включение/отключение способов оплаты: Telegram Stars / ЮKassa / Stripe / TON / VK Pay

- Управление каталогом игр: добавление, редактирование цены, назначение тарифу

### 14.3 Аналитика (дашборды)

- **Пользовательские метрики:** DAU/MAU, Retention (D1, D7, D30), Churn rate
- **Воронка:** Free → Basic → Plus → Pro по каналам и периодам
- **Финансы:** ARPU, LTV, CAC по источникам трафика, MRR, ARR
- **Продуктовые метрики:** топ режимов, популярные игры, % завершений квестов
- **Качество LLM:** thumbs up/down, средняя длина диалога, повторные вопросы
- **Проактивность:** % открытых инициативных сообщений, conversion rate
- **Adult Mode:** количество активированных (без персональных данных), churn/retention сравнение
- **Big Data срезы:** кластеры аудитории, тематические тренды, сезонность запросов

- Поиск по user_id, username, channel_user_id, email

- Просмотр (анонимизированных) жалоб на контент
- Экспорт данных пользователя в ZIP (по запросу пользователя, 152-ФЗ)
