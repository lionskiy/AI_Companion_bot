# Memory Architecture — Stage 2 Redesign Notes

## Текущая реализация (Stage 1)

### Уровни памяти

| Уровень | Хранилище | TTL | Что хранит |
|---------|-----------|-----|-----------|
| mem_L0 | Context Window (RAM) | duration of request | Последние N сообщений сессии из Redis |
| mem_L1 | Redis `session:{user_id}` | 48ч | session_id, created_at |
| mem_L2 | PostgreSQL `memory_episodes` + Qdrant `user_episodes` | permanent | Сжатые эпизоды диалогов (summary) |
| mem_L3 | PostgreSQL `memory_facts` + Qdrant `user_facts` | permanent | Факты о пользователе (извлечённые из эпизодов) |

Психологический профиль хранится в PostgreSQL `user_profiles` (JSONB поля), обновляется Celery-задачей `update_psych_profile`.

### Известные проблемы

**L1 → L2 (сжатие сессии в эпизод):**
- При сжатии теряется контекст: тон, эмоции, нюансы — LLM пишет "нейтральное резюме"
- Нет разделения на смысловые блоки внутри сессии — вся сессия = один эпизод
- Нет сигнала о важности: случайный small talk и кризисный момент сжимаются одинаково

**L2 → L3 (извлечение фактов):**
- Факты извлекаются из сжатого эпизода, не из оригинала → двойная потеря контекста
- Нет дедупликации: один и тот же факт может записаться несколько раз с разными формулировками
- Нет механизма устаревания: факт "хочет сменить работу" остаётся актуальным через год
- Нет confidence score: "точно знает" vs "возможно упоминал"

**RAG к knowledge_* коллекциям:**
- В текущем диалог-сервисе нет явного RAG к `knowledge_psych`, `knowledge_astro` и т.д. в момент ответа
- Знания используются только когда явно вызваны (tarot/astro хэндлеры), не в обычном чате

**Per-user индексация:**
- Все пользователи в одной Qdrant коллекции `user_episodes` и `user_facts`
- Фильтрация по `user_id` в payload — работает, но при большом scale создаёт hotspot
- Нет namespace isolation

**Психологический профиль:**
- Обновляется пакетно Celery-задачей, не real-time
- Профиль хранится как JSONB без версионирования — нельзя посмотреть "каким был профиль 3 месяца назад"
- Нет механизма "забыть" или "переосмыслить" устаревшие убеждения пользователя

## Что нужно сделать в Stage 2

### 1. Семантическое сжатие с сохранением эмоций

Заменить plain summary на структурированный эпизод:
```json
{
  "summary": "...",
  "emotional_tone": "anxious → relieved",
  "key_topics": ["работа", "отношения с мамой"],
  "significance": "high",
  "raw_turns_count": 12
}
```

Хранить оригинальные сообщения отдельно в PostgreSQL с TTL (например, 90 дней), после чего только summary.

### 2. Дедупликация и устаревание фактов

При записи нового факта — семантически сравнивать с существующими в Qdrant:
- similarity > 0.92 → обновить существующий, не создавать новый
- Добавить поля `first_seen_at`, `last_confirmed_at`, `confidence`
- Факты старше 180 дней без подтверждения → снижать confidence, помечать как "возможно устарел"

### 3. Явный RAG в диалоге

В `DialogService` при формировании контекста:
- Определять topic запроса (intent_classify)
- По topic делать RAG в соответствующую knowledge_* коллекцию
- Инжектировать топ-3 чанка в system prompt

### 4. Версионированный психологический профиль

Таблица `user_profile_history` — снапшоты профиля с датой. Позволяет:
- Отслеживать динамику (как менялся профиль за время)
- Откат если профиль деградировал
- Аналитика роста пользователя

### 5. Per-user Qdrant namespacing (при scale)

При достижении ~100k пользователей рассмотреть:
- Выделенные коллекции `user_episodes_{shard_id}` (шардирование по user_id range)
- Или Qdrant `sparse_vectors` + строгий filter (текущий подход, но с оптимизированными индексами)

### 6. Механизм "переосмысления"

Периодическая задача (раз в неделю) для активных пользователей:
- Перечитывает последние N эпизодов
- Выявляет противоречия в фактах
- Обновляет/удаляет устаревшие факты
- Обновляет psychological profile

## KB Organization (Этап 2+)

Вместо одной `knowledge_psych` — раздельные коллекции:

| Коллекция | Содержимое |
|-----------|-----------|
| `knowledge_psych_cbt` | КПТ: техники, протоколы, упражнения |
| `knowledge_psych_psychoanalysis` | Психоанализ, глубинная психология |
| `knowledge_psych_behavioral` | Поведенческая психология |
| `knowledge_psych_nlp` | НЛП: техники, паттерны |
| `knowledge_psych_articles` | Статьи, исследования (mixed topics) |
| `knowledge_stories` | Рассказы, сказки, нарративы |
| `knowledge_psych_datasets` | Датасеты диалогов для RAG |

Это позволяет делать таргетированный RAG: при CBT-запросе — только `knowledge_psych_cbt`, при нарративной технике — `knowledge_stories`.
